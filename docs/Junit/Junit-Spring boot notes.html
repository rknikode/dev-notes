Home / Introduction

1.1What is JUnit?

JUnit is a unit testing framework for Java. It is part of the xUnit family and has become the de facto standard for writing repeatable automated tests in Java. JUnit provides annotations to identify test methods, assertions to verify expected results, test runners to execute tests, and extensions to customize behavior.

Key Point

JUnit is not just a testing library — it is a complete testing platform supporting test discovery, execution, reporting, and extensibility through a modular architecture.

Java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

class CalculatorTest {
    @Test
    void shouldAddTwoNumbers() {
        Calculator calc = new Calculator();
        int result = calc.add(2, 3);
        assertEquals(5, result);
    }
}

Tests pass if all assertions are true, and fail otherwise. JUnit produces clear pass/fail reports with detailed failure messages.

Definition: Unit testing is the practice of testing individual units (methods, classes) of source code in isolation to verify they work as expected.

2006 — JUnit 4

2020+ — JUnit 5 becomes standard
Modern frameworks (Spring Boot, Gradle, Maven) adopt JUnit 5 as the default testing framework.

1.3Why Unit Testing?

Catches Bugs Early

Finds defects during development rather than in production, reducing cost and time.

Documentation

Tests serve as executable documentation of expected behavior and API usage.

Refactoring Safety

Gives confidence to refactor code without breaking existing functionality.

Faster Feedback

Immediate feedback on code changes within seconds, not hours.

Regression Prevention

Prevents reintroduction of previously fixed bugs.

Design Improvement

Encourages loosely coupled, testable designs through dependency injection.

  • Annotation-driven: Simple, declarative test definition with annotations.

1.5Testing Pyramid

The Testing Pyramid, introduced by Mike Cohn, describes the ideal distribution of automated tests:

E2E / UI TestsFew — Slow, Brittle
Integration TestsSome — Moderate speed
Unit TestsMany — Fast, Reliable
Principle
Write many fast, isolated unit tests; some integration tests; and few end-to-end tests. This maximizes confidence while keeping execution time low.

1.6AAA Pattern (Arrange-Act-Assert)

The AAA pattern is a standard way to structure unit tests into three logical sections:

Java
@Test
void testWithdraw() {
    // Arrange — Set up test data and objects
    BankAccount account = new BankAccount(1000);

    // Act — Execute the behavior under test
    account.withdraw(300);

    // Assert — Verify the result
    assertEquals(700, account.getBalance());
}
Output
Test passes because balance (1000 - 300 = 700) matches expected value.
Advantages: Clear separation of concerns, self-documenting tests, consistent structure across the project.

CHAPTER 2: JUnit 4

JUnit 4 introduced annotation-based testing and remains widely used in legacy and enterprise projects.

2.1Introduction to JUnit 4

Definition: JUnit 4 is a major revision of the JUnit testing framework released in 2006. It replaced the convention-based approach of JUnit 3 (which required extending TestCase and naming methods with "test" prefix) with a modern annotation-driven model using Java 5 annotations.

Definition: JUnit 4 uses annotations like @Test, @Before, @After, @BeforeClass, @AfterClass to mark test methods and lifecycle hooks. It introduced flexible test runners, custom Rules, and parameterized test support.
Why Upgrade from JUnit 3?

JUnit 4 eliminated boilerplate code (no more extends TestCase), enabled zero-argument constructors, allowed test methods to return void with any name, and introduced statically imported assertions for cleaner code.

Interview Note: Many legacy enterprise projects still use JUnit 4. Understanding it is crucial for maintenance work. JUnit 5's Vintage engine provides backward compatibility.
Java
// JUnit 3 style (old)
public class OldTest extends TestCase {
    public void testAdd() { assertEquals(5, 2 + 3); }
}

// JUnit 4 style (new, annotation-driven)
public class NewTest {
    @Test
    public void shouldAddTwoNumbers() { assertEquals(5, 2 + 3); }
}
Key Difference
JUnit 4: No base class required, annotations drive behavior, flexible method naming, static imports for cleaner assertions.

2.2JUnit 4 Architecture

Explanation: JUnit 4 is built around a TestRunner architecture. The TestRunner discovers test methods annotated with @Test, executes them, and reports results. The framework uses reflection to find and invoke methods, and manages the lifecycle through annotations.

Core Components

TestRunner: The engine that discovers and runs tests (e.g., BlockJUnit4ClassRunner). @Test annotation: Marks methods as test methods. Assert class: Static assertion methods. Rules: Reusable test behavior via TestRule interface. Runners: Custom test runners via @RunWith annotation.

How It Works: JUnit 4 uses Java reflection to scan for annotations. The runner identifies @Test, @Before, @After, @BeforeClass, @AfterClass methods, builds an execution order, and runs tests with reporting via JUnitCore or IDE integration.
Interview Note: Understanding the Runner architecture is key to understanding JUnit 4's limitations — only one @RunWith is allowed, which prevents combining features like parameterized tests with categories.

2.4Folder Structure

Definition: JUnit follows Maven's standard directory layout where test code mirrors the production code package structure.

Text
project-root/
  src/
    main/java/com/example/service/   // Production: UserService.java
    test/java/com/example/service/   // Tests: UserServiceTest.java
Naming Convention

Test classes typically use the suffix Test (e.g., CalculatorTest, UserServiceTest). Maven's Surefire plugin automatically picks up classes matching patterns like *Test.java, Test*.java, *TestCase.java.

Why This Structure? Mirroring the production package structure allows package-private access for test methods and makes it easy to locate corresponding tests. Maven and Gradle both expect this convention.

2.8Exception Testing

JUnit 4 provides three approaches to verify expected exceptions:

1. Using expected attribute in @Test

Java
@Test(expected = ArithmeticException.class)
public void testDivisionByZero() {
    int result = 1 / 0; // This will throw ArithmeticException
}
Limitation: Cannot verify the exception message or cause. Test passes if any part of the method throws the expected exception.

2. Using ExpectedException Rule

Java
@Rule
public ExpectedException thrown = ExpectedException.none();

@Test
public void testExceptionWithDetails() {
    thrown.expect(IllegalArgumentException.class);
    thrown.expectMessage("Invalid input");
    throw new IllegalArgumentException("Invalid input");
}

3. Try-Catch Pattern (Most Flexible)

Java
@Test
public void testExceptionWithCatch() {
    try {
        new Calculator().divide(1, 0);
        fail("Expected ArithmeticException to be thrown");
    } catch (ArithmeticException e) {
        assertEquals("/ by zero", e.getMessage());
    }
}
Output
All three approaches result in a PASS if the expected exception is thrown at the right point.

2.9@Ignore

Use @Ignore to temporarily disable a test without removing the code.

Java
@Ignore("Feature not yet implemented — will fix in next sprint")
@Test
public void testFutureFeature() {
    // This test will be skipped with the given message
}

@Ignore("Entire class is a work in progress")
public class WorkInProgressTest {
    // All tests in this class will be skipped
}
Best Practice: Use @Ignore sparingly. Ignored tests often stay ignored forever. Prefer fixing or deleting them. Always provide a reason for ignoring.

2.10@Test(timeout)

The timeout attribute specifies the maximum time (in milliseconds) a test should take.

Java
@Test(timeout = 1000) // Fails if test takes longer than 1 second
public void testPerformance() throws InterruptedException {
    assertTrue(performExpensiveOperation());
}

@Test(timeout = 100)
public void testWillTimeout() throws InterruptedException {
    Thread.sleep(500); // This will cause a failure
}
Real-world use: Timeout tests are critical for performance-sensitive operations like database queries, API calls, and file processing.

2.11Rules

Rules provide reusable test behavior. They implement TestRule interface:

Built-in RulePurpose

public class RuleExamplesTest {

    @Rule
    public TemporaryFolder tempFolder = new TemporaryFolder();

    @Rule
    public TestName testName = new TestName();

    @Rule
    public ErrorCollector collector = new ErrorCollector();

    @Test
    public void testRules() throws IOException {
        System.out.println("Test name: " + testName.getMethodName());

        File file = tempFolder.newFile("test.txt");
        assertTrue(file.exists());

        // ErrorCollector continues even if one assertion fails
        collector.checkThat("a", containsString("b")); // logs but doesn't stop
        collector.checkThat(1, is(2));                // logs but doesn't stop
    }
}

2.12Categories

Categories group tests for selective execution using marker interfaces:

Java
// Define marker interfaces
public interface SmokeTest {}
public interface IntegrationTest {}

// Apply categories to tests
public class CategorizedTest {
    @Category(SmokeTest.class)
    @Test
    public void smokeTest() { }

    @Category({SmokeTest.class, IntegrationTest.class})
    @Test
    public void integrationTest() { }
}

// Run only SmokeTest tests
@RunWith(Categories.class)
@IncludeCategory(SmokeTest.class)
@SuiteClasses(CategorizedTest.class)
public class SmokeTestSuite {}

2.14Parameterized Tests

Run the same test logic with different inputs using @RunWith(Parameterized.class):

Java
@RunWith(Parameterized.class)
public class MathParameterizedTest {

    @Parameters
    public static Collection<Object[]> data() {
        return Arrays.asList(new Object[][] {
            { 1, 2, 3 },   // a, b, expected
            { 2, 3, 5 },
            { 5, 5, 10 },
            { -1, 1, 0 }
        });
    }

    private int a, b, expected;

    public MathParameterizedTest(int a, int b, int expected) {
        this.a = a;
        this.b = b;
        this.expected = expected;
    }

    @Test
    public void testAddition() {
        assertEquals(expected, a + b);
    }
}
Output
4 test invocations: (1+2=3) ✓, (2+3=5) ✓, (5+5=10) ✓, (-1+1=0) ✓

2.15JUnit 4 Best Practices

Best Practices
  • Use meaningful names: test method names should describe the behavior being verified.
  • Follow AAA: Always structure tests as Arrange, Act, Assert.
  • One assertion per test: Each test should verify exactly one logical behavior.
  • Use @Before/@After: DRY up shared setup/cleanup code.
  • Isolate tests: Tests should never depend on other tests or execution order.
  • Parameterize: Use parameterized tests for testing multiple inputs.
  • Keep tests fast: Slow tests discourage execution. Aim for milliseconds.
  • Use assertThat: Hamcrest matchers produce more readable failure messages.
  • Test exception messages: Don't just check exception type, verify the message too.
  • Organize by package: Mirror production package structure in test source.

CHAPTER 3: JUnit 5 (Jupiter)

JUnit 5 is a complete rewrite with modular architecture, modern features, and powerful extensibility. It is the standard for new Java projects.

3.1Introduction to JUnit 5

JUnit 5 is the next generation of JUnit. It was designed from scratch to support modern Java features (Java 8+), provide better extensibility, and enable more powerful testing patterns. Unlike JUnit 4's monolithic JAR, JUnit 5 is composed of multiple modules.

Why JUnit 5?

Lambda assertions, grouped assertions, dynamic tests, nested tests, parameterized tests with rich sources, extensible extension model, parallel execution, and full backward compatibility with JUnit 4 via Vintage.

3.2Platform / Jupiter / Vintage

Three Sub-Projects

JUnit Platform: Foundation module. Provides the TestEngine API, Launcher API, and Console Launcher. Serves as the entry point for all test engines.

JUnit Jupiter: The new programming model. Contains junit-jupiter-api (annotations, assertions, assumptions) and junit-jupiter-engine (engine implementation).

JUnit Vintage: Provides a TestEngine for running JUnit 4 and JUnit 3 tests on the JUnit 5 platform. This ensures smooth migration.

3.6New Features in JUnit 5

Lambda Assertions

Lazy evaluation of error messages using lambdas. Only evaluates if assertion fails.

assertAll()

Group multiple assertions. All execute even if some fail.

assertThrows()

Returns exception for detailed verification of message, cause, etc.

Assumptions

Conditional test execution with assumeTrue, assumeFalse, assumingThat.

@ParameterizedTest

Multiple value sources: @ValueSource, @CsvSource, @MethodSource, etc.

@RepeatedTest

Repeat a test a specified number of times.

@Nested

Hierarchical test structure for better organization.

@DisplayName

Human-readable, descriptive test names with spaces and emojis.

@Timeout

Annotation-based timeout with support for methods, classes, and all tests.

Dynamic Tests

Runtime test generation via @TestFactory methods.

Extension Model

Powerful replacement for Rules and Runners.

Parallel Execution

Built-in parallel test execution with configuration options.

Java
import static org.junit.jupiter.api.Assertions.*;

@Test
void testJUnit5Features() {
    // Lambda assertion (lazy message evaluation)
    assertEquals(5, compute(2, 3),
        () -> "Expected 5 but got " + compute(2, 3));

    // Grouped assertions
    assertAll("user properties",
        () -> assertEquals("John", user.getFirstName()),
        () -> assertEquals("Doe", user.getLastName()),
        () -> assertNotNull(user.getEmail())
    );

    // Exception testing with assertThrows
    ArithmeticException ex = assertThrows(
        ArithmeticException.class,
        () -> { int x = 1 / 0; }
    );
    assertTrue(ex.getMessage().contains("zero"));
}

6.3Stubbing

Stubbing defines what a mock should return when a specific method is called:

Java
// Basic stubbing
when(userRepository.findById(1L)).thenReturn(new User(1L, "John"));

// Void methods
doNothing().when(emailService).sendEmail(any());

// Throw exception
when(userRepository.findById(eq(999L)))
    .thenThrow(new RuntimeException("User not found"));

// Multiple calls — different results
when(counter.getCount())
    .thenReturn(1)
    .thenReturn(2)
    .thenThrow(new RuntimeException());

// Answer interface — dynamic response
when(calculator.add(anyInt(), anyInt()))
    .thenAnswer(invocation -> {
        int a = invocation.getArgument(0);
        int b = invocation.getArgument(1);
        return a + b;
    });

6.4Verification

Verification checks that specific interactions happened with the mock:

Java
verify(userRepository).save(any(User.class));

// Verify exactly N times
verify(emailService, times(2)).sendWelcomeEmail(any());

// Verify at least/at most
verify(repository, atLeast(1)).findAll();
verify(repository, atMost(3)).save(any());

// Verify never called
verify(repository, never()).delete(any());

// Verify no more interactions
verifyNoMoreInteractions(repository);

// Verify order of calls
InOrder inOrder = inOrder(repository, emailService);
inOrder.verify(repository).save(any());
inOrder.verify(emailService).sendWelcomeEmail(any());

// Verify with timeout (for async code)
verify(asyncService, timeout(5000)).process();
Output
Verification passes if the expected interaction occurred; fails with a detailed error message showing actual invocations.

6.5@Spy

A spy wraps a real object. By default, it calls real methods, but can be stubbed selectively:

Java
@ExtendWith(MockitoExtension.class)
class SpyExampleTest {

    @Spy
    private List<String> list = new ArrayList<>();

    @Test
    void testSpy() {
        list.add("hello"); // real method called

        // Stub a specific method
        doReturn(100).when(list).size();

        assertEquals(100, list.size()); // stubbed
        assertEquals("hello", list.get(0)); // real
    }
}

6.6@Captor (ArgumentCaptor)

ArgumentCaptor captures method arguments for detailed assertions:

Java
@ExtendWith(MockitoExtension.class)
class CaptorExampleTest {

    @Mock private EmailService emailService;
    @InjectMocks private UserRegistration registration;

    @Captor
    private ArgumentCaptor<User> userCaptor;

    @Test
    void testCaptor() {
        registration.registerUser("john@test.com", "John");

        assertNotNull(capturedUser.getRegistrationDate());
    }
}

CHAPTER 9: Quick Revision

Common mistakes, best practices, and a concise summary for last-minute review before interviews.

9.1Common Mistakes

  1. Tests depending on order: Never rely on test execution order. Use @BeforeEach for setup.
  2. Not testing edge cases: Empty strings, null values, boundary conditions are often missed.
  3. Testing multiple things in one test: Each test should verify one behavior. Use assertAll() for grouping.
  4. Ignoring exception messages: Use assertThrows() to verify both exception type and message.
  5. Using Thread.sleep() in tests: Use awaitility or timeout instead of fixed delays.
  6. Forgetting to mock dependencies: Unit tests should isolate the class under test.
  7. Not cleaning up resources: Files, DB connections, etc. should be cleaned up in @AfterEach.
  8. Over-mocking: Mocking simple value objects leads to brittle tests.
  9. Testing private methods: Test through public APIs. Private methods are implementation details.
  10. Using real databases in unit tests: Use mocked repositories or in-memory databases.
  11. Not using parameterized tests: Copy-pasting the same test with different inputs is wasteful.
  12. Ignoring flaky tests: Flaky tests undermine confidence. Fix or remove them.

9.2Best Practices Summary

  • Follow AAA pattern: Arrange, Act, Assert
  • Use meaningful test names: shouldReturnUserWhenIdIsValid
  • Write tests first (TDD) for better design
  • Keep tests independent — no shared mutable state
  • Use @BeforeEach for setup, @AfterEach for cleanup
  • Prefer JUnit 5 over JUnit 4 for new projects
  • Use @ParameterizedTest for data-driven tests
  • Use @MethodSource for complex test data
  • Use assertAll() for multiple related assertions
  • Use assertThrows() for exception testing
  • Use @DisplayName for descriptive test names
  • Use @Tag to categorize tests (unit, integration, slow)
  • Use Mockito wisely — mock external dependencies only
  • Use @ExtendWith(MockitoExtension.class) in JUnit 5
  • Use BDDMockito (given/willReturn/then) for readability
  • Cover boundary conditions, edge cases, and error paths
  • Keep tests fast — use in-memory DBs and mocked networks
  • Run tests frequently — integrate with CI/CD pipeline
  • Review test coverage but don't chase 100% blindly

What is JUnit and why is it used?
▼
Easy

▼
Easy

3
What is the AAA pattern in unit testing?
▼
Easy

What is the difference between @BeforeEach and @BeforeAll?
▼
Easy

▼
Easy

What is the difference between assertEquals and assertSame?
▼
Easy

What does @Disabled do in JUnit 5?
▼
Easy

▼
Easy

Answer: FIRST stands for: Fast (tests run quickly), Isolated (no dependencies between tests), Repeatable (same result every time), Self-validating (pass/fail clearly), Timely (written alongside production code).

What is Mockito and why would you use it with JUnit?
▼
Easy

Answer: Mockito is a mocking framework that creates mock objects for dependencies. ▼

What is the difference between Mock and Spy in Mockito?
▼
Medium

How does @Nested work in JUnit 5?
▼
Medium

15
What is assertAll() and when should you use it?
▼
Medium

Answer: assertAll() groups multiple assertions so that all are executed even if some fail. It reports all failures together rather than stopping at the first failure. Use it when validating multiple properties of an object (e.g., checking all fields of a User after creation).

▼
Medium

Answer: Assumptions (imported from org.junit.jupiter.api.Assumptions) allow conditional test execution. If an assumption fails, the test is aborted (not failed). Examples: assumeTrue(osName.startsWith("Windows")) — test only runs on Windows. Useful for environment-specific tests.

▼
Medium

Answer: The Testing Pyramid describes the ideal distribution of tests: many unit tests (fast, isolated), some integration tests, and few end-to-end tests. This maximizes test confidence while keeping execution time low. Introduced by Mike Cohn.

What is @SpringBootTest and what does it do?
▼

Explain the JUnit 5 Extension model. How is it different from JUnit 4 Rules?
▼
Hard

Answer: Extensions replace JUnit 4's @Rule and @RunWith. They implement callback interfaces (BeforeEachCallback, AfterEachCallback, ParameterResolver, etc.) and are registered via @ExtendWith. ▼
Hard

▼
